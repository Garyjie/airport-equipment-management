## 为什么需要开发流程？

### 🍳 用做饭打个比方

想象你要做一桌年夜饭：

- **没有流程的人**：想到什么做什么 → 菜炒到一半发现没买盐 → 汤炖上了才发现锅不够 → 最后手忙脚乱，菜要么凉了要么糊了
- **有流程的人**：先列菜单 → 买菜 → 备菜 → 按顺序炒菜 → 摆盘 → 上菜 → 每一步都井井有条

开发也是一样的道理。**没有流程的开发 = 边想边写边改 = 漏洞百出 = 无限返工**。

### 这个项目的教训

回顾咱们的机场设备管理系统，走了不少弯路：

| 问题 | 原因 | 如果按流程来会怎样 |
|------|------|-------------------|
| 页面崩溃 | 前后端数据格式不一致 | 设计阶段就约定好数据格式 |
| 添加设备失败 | 外键约束没考虑到 | 设计数据库时就想到空值处理 |
| 导入没反应 | 编码问题、重复问题没考虑 | 需求阶段就列出导入的所有边界情况 |
| 弹窗关不掉 | 没有错误处理 | 编码阶段就强制 try-catch |

**好的流程，就是把坑提前想清楚，而不是踩了才填。**

## 传统开发流程（7 步法）

传统开发就像建房子，一步一步来，每一步都有明确的交付物。

```
需求分析 → 设计 → 编码 → 测试 → 部署 → 运维 → 迭代
   ①       ②      ③      ④      ⑤      ⑥      ⑦
```

### 第 1 步：需求分析（想清楚做什么）

**目标**：把"我想做个 XX 系统"变成"具体要做哪些功能"。

**做什么**：
1. 写一份需求文档（PRD）
2. 列功能清单（可以用思维导图）
3. 划分优先级（哪些必须有，哪些可以后加）
4. 画原型图（纸上画也行，用 Figma 也行）

**交付物**：需求文档 + 功能清单 + 原型草图

**举例 - 机场设备管理系统需求**：
```
核心功能：
  必做（P0）：
    - 用户登录/登出
    - 设备的增删改查
    - 设备状态管理（正常/损坏/维修中）
  重要（P1）：
    - 站点管理
    - 换纸记录
    - 数据导入导出
  可选（P2）：
    - 仪表盘统计
    - 操作日志
    - 消息通知
```

**新手提示**：
- ✅ 需求一定要写下来，别只在脑子里想
- ✅ 第一版功能越少越好，做减法比做加法难
- ❌ 别一上来就想"我要做一个完美的系统"

---

### 第 2 步：设计（想清楚怎么做）

**目标**：在写代码之前，把架构、数据结构、接口都设计好。

**做什么**：

#### 2.1 技术选型
```
前端：React / Vue / Next.js？
后端：Express / Koa / Nest.js？
数据库：MySQL / PostgreSQL / SQLite？
```

#### 2.2 数据库设计（最最重要！）
- 画 ER 图（实体关系图）
- 确定每张表的字段、类型、约束
- 想好表之间的关系（一对一、一对多、多对多）

#### 2.3 API 接口设计
- 约定 RESTful 风格
- 每个接口的请求参数、返回格式
- 统一的响应格式

#### 2.4 页面结构设计
- 页面路由规划
- 组件拆分思路

**交付物**：数据库设计文档 + API 接口文档 + 页面路由图

**举例 - 设备表设计**：
```
Device 表：
  id: 主键，自增
  name: 设备名称，字符串，必填
  serialNumber: 设备编号，字符串，唯一，必填
  typeId: 设备类型ID，外键 → DeviceType
  stationId: 站点ID，外键 → Station，可以为空
  status: 状态（normal/damaged/maintaining）
  createdAt: 创建时间
  updatedAt: 更新时间
```

**新手提示**：
- ✅ 数据库设计花多少时间都不为过，改表结构的成本远高于改代码
- ✅ 接口格式要前后端一起约定好，写下来
- ❌ 别跳这步，"边写边想"是新手最大的坑

---

### 第 3 步：编码（动手实现）

**目标**：按照设计，把代码写出来。

**推荐顺序（从底层到上层）**：
```
数据库表 → 后端 API → 前端页面 → 联调
```

**做什么**：
1. 搭建项目骨架（目录结构、配置文件）
2. 写数据库模型
3. 写后端 API 接口
4. 写前端页面
5. 前后端联调

**编码原则**：
- 一个功能一个功能地做，做完一个测一个
- 写注释（给自己看，也给未来的自己看）
- 变量名/函数名要见名知意

**新手提示**：
- ✅ 先让功能跑通，再优化代码（先跑起来，再变好）
- ✅ 每完成一个小功能就测试一下，别攒一大堆再测
- ❌ 不要追求"完美的代码"，第一版能工作就行

---

### 第 4 步：测试（找 Bug）

**目标**：在用户发现之前，自己把问题找出来。

**测试类型**：

| 测试类型 | 说明 | 新手怎么做 |
|---------|------|-----------|
| 单元测试 | 测单个函数/组件 | 可以先不做，后面补 |
| 集成测试 | 测模块之间的配合 | 重点做！测 API 接口 |
| 功能测试 | 像用户一样点一遍 | 必须做！每个按钮都点一遍 |
| 边界测试 | 测极端情况 | 空值、超长、重复... |

**测试清单举例**：
```
设备管理页面：
  □ 添加设备 - 正常情况
  □ 添加设备 - 编号重复会怎样
  □ 添加设备 - 不填必填项会怎样
  □ 编辑设备 - 修改后保存成功
  □ 删除设备 - 确认弹窗正常
  □ 搜索设备 - 能搜到/搜不到的情况
  □ 导入设备 - 正常文件
  □ 导入设备 - 空文件
  □ 导入设备 - 格式错的文件
```

**新手提示**：
- ✅ 把自己当"找茬的"，专门想"怎么用会出错"
- ✅ 边界情况比正常情况更容易出 Bug
- ❌ 别觉得"我写的代码没问题"，没有人写的代码没 Bug

---

### 第 5 步：部署（上线）

**目标**：让别人能访问你的系统。

**做什么**：
1. 买服务器（或用免费的：Vercel、Railway 等）
2. 配置生产环境
3. 部署数据库
4. 部署后端
5. 部署前端
6. 配置域名（可选）

**新手提示**：
- ✅ 第一次部署用平台托管（Vercel、Netlify），别自己搭服务器
- ✅ 部署后再测一遍，生产环境和本地可能不一样
- ❌ 别在代码里写死密码、密钥！用环境变量

---

### 第 6 步：运维（运行维护）

**目标**：系统上线后，保证它稳定运行。

**做什么**：
- 监控系统状态
- 备份数据库
- 处理用户反馈的问题
- 看日志排查错误

**新手提示**：
- ✅ 日志很重要，出问题了全靠它
- ✅ 数据库定时备份，万一崩了能恢复
- ❌ 别上线了就不管了

---

### 第 7 步：迭代（持续改进）

**目标**：根据用户反馈，不断完善功能。

**做什么**：
1. 收集用户反馈
2. 排优先级
3. 下一轮开发
4. 重复上面的流程

**新手提示**：
- ✅ 小步快跑，每次更新一点点
- ✅ 每次更新都要有记录（版本号）

## Vibe Coding 流程（AI 辅助开发 9 步法）

Vibe Coding 就是"跟着感觉写代码"，但不是瞎写，而是**用 AI 当助手，快速验证想法**。

它的核心是：**先让 AI 搭出骨架，人来把控方向和细节**。

```
想法 → 拆解 → AI生成骨架 → 人工调整 → AI填充细节 → 人工验证 → 测试 → 修复 → 完成
 ①     ②        ③          ④          ⑤          ⑥        ⑦      ⑧      ⑨
```

### 第 1 步：想法（Vibe 阶段）

**目标**：把模糊的想法说清楚。

**做什么**：
- 用一句话描述你要做什么
- 想清楚给谁用、解决什么问题
- 列 3-5 个核心功能

**提示词模板**：
```
我想做一个[XX系统/工具]，主要给[目标用户]使用，用来解决[什么问题]。
核心功能有：
1. xxx
2. xxx
3. xxx

请你帮我评估一下这个想法的可行性，并且给出技术选型建议。
```

**举例**：
```
我想做一个机场设备管理系统，主要给运维人员使用，用来管理机场里的各种设备（电脑、打印机、扫描仪等），记录设备状态和维护记录。
核心功能有：
1. 设备的增删改查
2. 设备状态管理（正常/损坏/维修中）
3. 站点和柜台管理
4. 换纸记录
5. 数据导入导出

请你帮我评估一下这个想法的可行性，并且给出技术选型建议。
```

---

### 第 2 步：拆解（拆成小任务）

**目标**：把大项目拆成 AI 能一次完成的小任务。

**拆任务的原则**：
- 每个任务能在一次对话内完成
- 从底层到上层（数据 → 后端 → 前端）
- 先搭骨架，再填肉

**拆解举例 - 机场设备管理系统**：
```
第一阶段：项目搭建
  任务1：初始化项目结构（Next.js + Express + SQLite）
  任务2：设计数据库模型

第二阶段：后端开发
  任务3：实现用户认证 API（登录/注册）
  任务4：实现设备类型 CRUD API
  任务5：实现站点管理 CRUD API
  任务6：实现设备管理 CRUD API
  任务7：实现记录相关 API

第三阶段：前端开发
  任务8：搭建前端布局和路由
  任务9：实现登录页面
  任务10：实现设备管理页面
  ...以此类推
```

**提示词模板**：
```
我要做一个[XX系统]，请帮我把开发任务拆解成 15-20 个小任务，
按照从易到难、从底层到上层的顺序排列。
每个任务要具体到能直接动手做的程度。
```

---

### 第 3 步：AI 生成骨架（让 AI 先写第一版）

**目标**：快速生成可运行的基础版本。

**做什么**：
- 把需求告诉 AI
- 让 AI 生成完整的代码
- 跑起来看看效果

**提示词模板**：
```
请帮我实现[具体功能]。

技术栈：
- 前端：Next.js + TypeScript + TailwindCSS
- 后端：Express.js
- 数据库：SQLite + Prisma

需求详情：
1. [需求1]
2. [需求2]
3. [需求3]

要求：
- 代码要完整，可以直接运行
- 要有注释，方便我理解
- 错误处理要完善
- 遵循 RESTful API 规范

请生成所有需要的文件，并说明每个文件的作用。
```

**新手提示**：
- ✅ 一次只让 AI 做一件事，别一次塞太多需求
- ✅ 需求描述越清楚，AI 写得越好
- ❌ 别完全相信 AI 生成的代码，一定要自己看一遍

---

### 第 4 步：人工调整（人来掌舵）

**目标**：修正 AI 生成的代码，让它符合你的预期。

**常见需要调整的地方**：
- 技术细节不对（比如你用的是 Koa 不是 Express）
- 字段名和你想的不一样
- 缺少某些功能
- 代码结构不符合你的习惯

**怎么做**：
- 先读 AI 生成的代码
- 标记出不对的地方
- 一个个告诉 AI 怎么改

**提示词模板**：
```
刚才生成的代码我看了，有几个地方需要调整：

1. [问题1] → 改成[什么样子]
2. [问题2] → 改成[什么样子]
3. [问题3] → 改成[什么样子]

请修改对应的文件。
```

---

### 第 5 步：AI 填充细节（逐步完善）

**目标**：在骨架基础上，添加更多功能和细节。

**做什么**：
- 一个功能一个功能地加
- 每次加完就测试
- 发现问题马上改

**提示词模板 - 添加功能**：
```
现在我要给[某个页面/模块]添加[某个功能]。

具体需求：
- [详细描述]
- [边界情况1]怎么处理
- [边界情况2]怎么处理

请修改相关文件。
```

**提示词模板 - 修复问题**：
```
刚才的代码运行有问题：
- 现象：[描述现象]
- 错误信息：[粘贴错误信息]
- 我期望的行为：[描述应该怎样]

请帮我找出原因并修复。
```

---

### 第 6 步：人工验证（人来把关）

**目标**：确保每个功能都按预期工作。

**做什么**：
- 跑起来，手动点一遍
- 测正常情况
- 测边界情况（空值、重复、超长...）
- 看代码，确保逻辑是对的

**验证清单**：
```
□ 功能跑通了吗？
□ 正常操作有没有问题？
□ 出错了有没有提示？
□ 代码逻辑我看懂了吗？
□ 有没有明显的 Bug？
```

**新手提示**：
- ✅ 一定要自己跑一遍，别 AI 说"好了"就信了
- ✅ 看不懂的代码一定要问 AI 解释
- ❌ 别跳过验证，坑都藏在你没测的地方

---

### 第 7 步：测试（系统性找 Bug）

**目标**：像真实用户一样全面测试。

**做什么**：
- 把所有功能点一遍
- 故意用错的方式操作
- 测各种边界情况

**测试思路**：
```
正常流程测试：按正确的操作顺序走一遍
异常流程测试：故意点错、输错、跳过步骤
边界值测试：最大值、最小值、空值、特殊字符
数据一致性测试：改了一个地方，相关的数据有没有同步更新
```

---

### 第 8 步：修复（解决发现的问题）

**目标**：把测试发现的 Bug 都修好。

**怎么做**：
1. 列出来所有发现的问题
2. 按严重程度排序
3. 一个个让 AI 帮忙修
4. 修完再测一遍

**提示词模板**：
```
我发现了几个 Bug，帮我按顺序修一下：

Bug 1：
  现象：[描述]
  复现步骤：1. xxx 2. xxx 3. xxx
  期望：[应该怎样]

Bug 2：
  ...

请逐个修复，每修复一个就告诉我修复了什么。
```

---

### 第 9 步：完成 + 复盘

**目标**：总结经验，下次做得更好。

**做什么**：
- 写一份项目总结
- 记录遇到的坑和解决方案
- 想想哪些地方可以改进

**复盘模板**：
```
这次项目：
  做得好的地方：
    1. xxx
    2. xxx
  
  踩过的坑：
    1. xxx → 原因是 xxx → 下次应该 xxx
    2. xxx → 原因是 xxx → 下次应该 xxx
  
  下次改进：
    1. xxx
    2. xxx
```

## 两种方式对比

| 维度 | 传统开发 | Vibe Coding（AI 辅助） |
|------|---------|----------------------|
| **速度** | 慢（一步步来） | 快（AI 帮你写大量代码） |
| **质量** | 高（想清楚再写） | 中等（需要人来把关） |
| **适合人群** | 有经验的开发者 | 新手 / 快速验证想法 |
| **学习效果** | 好（每一步都理解） | 中等（容易依赖 AI） |
| **出错率** | 低（设计阶段排除问题） | 中高（需要大量测试） |
| **适合项目** | 大型项目 / 企业项目 | 小型项目 / 原型验证 |

### 💡 新手建议

**新手最佳实践 = Vibe Coding 的速度 + 传统开发的严谨**

具体来说：
1. **需求和设计阶段**：用传统开发的思路，想清楚再写
2. **编码阶段**：用 AI 辅助，提高效率
3. **测试阶段**：用传统开发的方法，仔细测
4. **复盘阶段**：两种方式都要总结

## Vibe Coding 提示词技巧大全

### 写好提示词的 5 个原则

#### 原则 1：越具体越好
❌ 不好的："帮我写个登录页面"
✅ 好的：
```
帮我写一个登录页面，要求：
- 使用 Next.js + TypeScript + TailwindCSS
- 有用户名和密码两个输入框
- 有"记住我"复选框
- 有登录按钮，加载时显示 loading 状态
- 表单验证：用户名不能为空，密码至少 6 位
- 调用 POST /api/auth/login 接口
- 登录成功跳转到 /dashboard
- 登录失败显示错误提示
```

#### 原则 2：给出上下文
告诉 AI 你的项目背景、技术栈、已有的代码结构。

```
这是一个机场设备管理系统，技术栈是 Next.js + Express + Prisma + SQLite。
目前已经实现了用户认证和设备管理功能。
现在我要添加一个换纸记录的功能...
```

#### 原则 3：分步进行
别一次塞一大堆需求，拆成小步骤。

```
第一步：先设计数据库模型
第二步：再写后端 API
第三步：最后写前端页面
```

#### 原则 4：给出示例
给 AI 看你想要的格式/风格。

```
返回格式要像这样：

  success: true,
  data: { ... },
  message: "操作成功"
}
```

#### 原则 5：让 AI 解释
看不懂的地方，让 AI 解释给你听。

```
请用初学者能听懂的话解释一下这段代码的逻辑，
最好打个比方。
```

## 新手常见坑与避坑指南

### 坑 1：需求不清就开始写

**表现**：写着写着发现"哦，我还要加个 XX 功能"，改来改去。

**避坑**：
- 写需求文档，哪怕只有一页
- 第一版功能越少越好，先做核心功能
- 想清楚再加功能，别边写边加

### 坑 2：数据库设计草率

**表现**：写代码时发现"哎呀，这个字段忘了加"、"那个关系设计错了"，改表改半天。

**避坑**：
- 数据库设计花 2-3 天想清楚
- 把所有实体和关系列出来
- 考虑未来可能的扩展

### 坑 3：不做错误处理

**表现**：正常操作没问题，一出错就白屏、弹窗关不掉、没反应。

**避坑**：
- 每个 API 请求都加 try-catch
- 每个用户操作都要有反馈（成功/失败提示）
- 想清楚"出错了怎么办"

### 坑 4：不测边界情况

**表现**：正常用没问题，用户一用就出 Bug（空值、重复、超长字符...）

**避坑**：
- 测试时专门想"怎么用会出错"
- 列测试清单，一条条测
- 导入导出功能要测各种格式的文件

### 坑 5：依赖 AI，自己不理解

**表现**：AI 写的代码看不懂，出了问题不知道怎么改。

**避坑**：
- AI 写的代码自己要读一遍
- 不懂的地方让 AI 解释
- 关键逻辑自己写一遍

### 坑 6：不写注释和文档

**表现**：过了一个礼拜，自己写的代码都看不懂了。

**避坑**：
- 复杂的逻辑写注释
- 每个模块是干什么的，记下来
- API 接口文档要维护

### 坑 7：想一次做完美

**表现**：第一个版本就想做所有功能，结果越做越乱，哪个都做不好。

**避坑**：
- 先做最小可用版本（MVP）
- 功能一个一个加
- 小步快跑，持续迭代

## 🎯 最后给新手的几句话

1. **慢慢来，比较快** — 前期多花时间想清楚，后期少花时间改 Bug
2. **先跑起来，再变好** — 别追求第一版就完美，能工作是第一位的
3. **Bug 是正常的** — 没有人写的代码没 Bug，发现 Bug 是进步的机会
4. **AI 是助手，不是替代品** — 用 AI 提高效率，但别让 AI 替你思考
5. **写文档不是浪费时间** — 现在花 10 分钟写，未来可能省 10 个小时
6. **复盘很重要** — 每做完一个项目，总结经验教训，进步才快
